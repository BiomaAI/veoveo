Autonomy Harness

A self-hosted MCP gateway, a family of hosted tool servers, an agent kernel, and a durable recording hub for regulated workloads. Long-lived autonomous agents — the Pilot is the first — work in bounded episodes over one public endpoint that carries the full MCP protocol through enterprise auth and typed policy to domain servers, all sharing a single policy-enforced artifact plane. Every sensor, agent, and simulator streams its time-and-space data into one durable, queryable record. A live SUMO traffic world proves the whole loop end to end.

Component Reference

What every box is

Each named service, protocol, and format above, defined by what it is and what it does. The tag on each says its role: ours (we built it), OSS (an open-source component we run), protocol (a wire language), format (a data shape on disk or on the wire), client (something a person or agent attaches), or concept (an idea the system is built on).

Access & control

Edgeours

The single public door. One internet-facing endpoint (fronted by Cloudflare) that terminates TLS and absorbs abuse.

Every request from outside enters here and nowhere else, so there is exactly one place to defend and observe.

MCP Gatewayours

The front desk for tools. It authenticates the caller, checks policy, renames each server's tools into one flat menu, and writes an audit trail.

Anything an agent or client can do passes through it, so access rules live in one place instead of in every server.

Agent Kernelours

The runtime for long-lived agents. It runs an agent in short episodes: assemble what it needs to know, let it act, then sleep.

Memory lives in files, not chat history, so an agent can run for weeks, survive a restart, and wake on a result, a timer, or a message.

Recording Hub — the durable record

Rerun proxyOSS

The ingest socket. A gRPC server (re_grpc_server) where every producer pushes data, and which rebroadcasts it live to anyone reading.

It is the single meeting point: producers write to it, viewers read from it. It holds only a short in-memory buffer — durability is the spooler's job.

hub-spoolerours

Catch-and-persist. Our service that embeds the Rerun proxy and writes every message it receives to disk before anything else.

Because the proxy and the writer are one process, no data can arrive and be lost before it is saved — the write is the first thing that happens, not an afterthought.

Segment file .rrdformat

The record on disk. Rerun's recording format — one growing file per dataset / day / recording.

It stays readable even mid-write, so a crash leaves a decodable file up to the last whole message. This is the durable truth; everything else is a view onto it.

hub-catalogours

The reading room. Our service (running Rerun's rerun server) that serves the spooled files for querying over the network.

A headless data server: it answers questions about recordings, it does not draw anything.

redapprotocol

How you ask the hub for data. Rerun's remote data-access protocol — a query API for recordings, spoken over gRPC.

It lets a client pull just the slice it needs — latest-at (every vehicle's most recent position), range (one vehicle's path over five minutes), or a full dataframe — without downloading the whole file.

dataset · recordingconcept

How the record is organised. A recording is one stream (one SUMO run, one agent's decision log); a dataset groups them (world, agents).

In the catalog a segment's id is its recording id, so the file on disk and the thing you query are the same identity.

Rerun viewerclient

The window that draws it. A GUI app that renders the streams — vehicles on a map, speeds as graphs, a scrubbable timeline.

It attaches to the hub like any other reader: to the proxy for a live tail, or to a .rrd file to replay a whole run. It runs where you are, not in the containers.

GeoPoints · Scalarsformat

Typed data shapes. Rerun archetypes — GeoPoints are points with real lat/lon that land on a map; Scalars are named time-series numbers.

Logging data as a known shape is what lets the viewer put vehicles on a map and plot speeds without any per-project wiring.

SUMO Showcase — the live proof

SUMOOSS

The traffic simulator. An open-source microsimulator (Eclipse SUMO) that steps a road network full of individual vehicles through time.

It stands in for a real fleet of sensors: a live world that produces a stream of positions and states to record and reason about.

TraCIprotocol

The socket into the sim. SUMO's Traffic Control Interface — a connection to advance the simulation one step and read or command it.

Through it you get every vehicle's position and speed, and you can change a traffic light or reroute a car mid-run.

sumo-mcpours

The bridge. A Python service that holds the one TraCI connection, pushes the world into the hub, and exposes SUMO control as governed sumo__* tools.

One process is the sole owner of the sim, so reads, pushes, and commands never collide.

MCPprotocol

The tool language. Model Context Protocol — the standard the gateway, the servers, and the agents all speak for calling tools, reading resources, and running tasks.

Because every server speaks it, the gateway can front them all through one endpoint with one set of rules.

MCP taskconcept

A tool call you walk away from. A long operation returns a task id immediately; the agent sleeps and is woken when the result is ready.

This is how an agent runs a 50-step batch or a signal optimisation without holding a connection open or burning an episode waiting.

Resource · subscribeconcept

A condition that wakes you. An MCP resource is a readable endpoint; subscribing to one means the agent is notified the moment it changes.

The showcase exposes sim://congestion, so a jam forming is a wake, not something the agent has to poll for.

Agent Kernel

long-lived autonomous agents in bounded episodes — the Pilot is the first

An agent runs forever because chat history is never the memory: every episode's context is assembled fresh from the agent's two memory planes, and everything in flight survives process death. The kernel executes typed manifests — an agent type is a manifest, a preamble, and SQL migrations, never kernel code.

Wake

A batch of wakes, coalesced and deduped; every wake is persisted with its disposition, so the ledger explains why each episode ran.

Assemble

State header + the wake payload + manifest-defined SQL sections over the memory database, under a token budget. Truncations point at memory_query.

Episode

One bounded LLM run. Quick gateway tools answer inline; long tools dispatch as MCP tasks and keep running. Hooks record every step and terminate over-budget runs.

Persist

Unresolved task descriptors (serializable) land in the ledger; the episode closes with usage counters and a summary; every step is already in the decision log.

Sleep

Idle on the wake bus. A watcher per detached task holds the blocking tasks/result long-poll; the heartbeat bounds silence.

↩ wakes: task completion push · operator message · heartbeat · resource update · elicitation answer — and a wake survives process death: descriptors rehydrate from the ledger on any boot, so kill and restart resumes the same tasks.
Two memory planes
  • Current truth — an embedded relational database: domain tables (the Pilot: targets, missions, waypoints, constraints, beliefs) plus the kernel ledger of episodes, tasks, and wakes.
  • Decision log — an append-only Rerun recording: every turn, tool call, task lifecycle, prompt, and summary, time-indexed and watchable live in the Recording Viewer.
  • The agent reads both itself: memory_query (guarded SQL), memory_write (typed mutations, mirrored to the decision log), and timeline_query (dataframe reads over the log).
  • The decision log is the log of record: replay rebuilds the domain tables from it alone, so plane drift is recoverable.
Durability, proven
  • Detach at episode end: a task's descriptor outlives the process; any later boot rehydrates the live handle and keeps waiting.
  • Token rotation is make-before-break; task ids are principal-scoped, so continuity holds across reconnects.
  • The wake is push, arithmetically: with polling pinned at 60s and the heartbeat at 600s, the smoke's agent slept the task's full ~15s and woke within a second of completion — verified live with a real model end to end.
  • Budget hooks book overruns as budget_terminated, a first-class outcome; hourly windows defer non-priority wakes.
Agents are data
  • A typed manifest, validated fail-closed at boot: model endpoint, gateway principal (client-credentials + private-key JWT), episode limits, budgets, schedule, context SQL sections, writable tables.
  • A preamble and domain migrations complete the profile — cross-episode instructions live here or in memory, never in a prior prompt.
  • Operators reach a running agent over a loopback channel: inject a prompt, read status, answer parked tool questions (elicitations).
  • The Pilot rides the operator profile: geodesy inline, planning as tasks, waypoints recorded when plans land.

Shared Planes

the shared foundation every server builds on

Artifact Plane

A single byte-level policy-enforcement point. Every domain server stores and reads artifact bytes through the plane, calling it with the gateway-signed identity it already holds.

  • Per-tenant encrypted object store + Postgres grant ledger
  • Every read/write decided by the shared access contract: principal, tenant, data label, grant
  • Immutable content addressing: {scheme}://artifact/{sha256}
  • artifact:// references let one server consume another's output — e.g. SQL ingesting a media render, optimization snapshotting to a forecastable table

Contract Core

The typed, provider-neutral contract every server shares, validated fail-closed at startup.

  • Strong ID newtypes: profiles, servers, tools, tenants, scopes, JWT ids
  • Gateway control-plane, auth, and policy config with cross-reference validation
  • Typed deployment profiles: object store, state store, telemetry, network boundaries, regulated-data controls
  • Shared task, artifact, usage, and analytics schemas; JSON Schema export for other languages
  • Gateway-to-server internal signed identity assertions

Recording Hub

the durable, queryable time-and-space record — one bus, many producers

Every sensor, estimator, and agent streams Rerun data into one gRPC ingest point. A spooler embeds the proxy and persists every message to day-partitioned segment files — proxy and durable writer in one process, so the write is first-class and no ring-buffer drop can lose data. Files are footer-less while live (crash-decodable) and gain a manifest at freeze; the OSS Rerun catalog serves the same bytes over redap. Files are the record, the proxy is the bus, the catalog is the reading room.

Producers → bus
  • Many SDKs stream into one proxy, sessioned by recording id.
  • Routing: application-id prefix → dataset (world, agents, …); unmatched → quarantine.
  • Internal-network only — the OSS proxy trusts every caller.
Durability
  • Segment per (dataset, day, recording).
  • Footer-less live → decodable to the last whole message after a crash.
  • Freeze runs rerun rrd optimize for a lazy-loadable manifest.
Reading room
  • Local QueryEngine is the authoritative file reader.
  • Catalog serves the same files over redap; segment id == recording id.
  • latest-at = a world snapshot · range = a trajectory.

SUMO Showcase

a live traffic world proving perceive → decide → act, end to end

The SUMO traffic simulator runs a live city in its own container. A single Python sumo-mcp server owns the one TraCI connection, pushes the world into the hub as typed /world/sumo/** streams, and exposes SUMO control as governed sumo__* tools. The long operations are MCP tasks the agent detaches from and wakes on — the same durable sleep/wake the kernel already runs, now driven by a real simulator. SUMO is just another hub producer; the hub never pulls.

Why our own, in Python
  • It's what clients build — a proper task-native server, not a stdio child.
  • Full task API, gateway auth/policy, resource subscriptions, pydantic types.
  • Task line pinned (mcp 1.28.x, SEP-1686) behind one migration seam.
Tools
  • sync: query_state, describe_scenario, set_signal_phase, reroute_vehicle.
  • tasks: run_batch, generate_network, compute_routes, optimize_signals.
  • events: sim://congestion — subscribe → wake.
One sim owner
  • TraCI is single-client; one asyncio owner steps, pushes, and applies commands.
  • Push loop and tools share the toolset lock — no interleaving.
  • Vehicles as GeoPoints, speeds and phases as scalars.
