Capabilities

What the platform does

Each capability below is live, smoke-tested behavior of the current installation.

Detect, decide, intervene — nonstop

Agents chase the live picture around the clock: they catch conditions the moment they change, decide, and intervene — retiming signals, rerouting vehicles, retasking work — and between events they drive the mission forward on their own initiative. The tempo holds through restarts, upgrades, and machine failures, memory and in-flight work intact.

Dispatch long work and move on

A render, a forecast, a plan, or a 500-step simulation batch carries its own guarantees: it completes on its own, delivers the result the moment it is ready, and an interruption has a defined, recorded outcome.

Build a world model of the operation

Every sensor, memory, and agent decision fuses into one durable world model — ground truth for agents in the moment, an evidence chain after the fact. Any run replays, any moment is queryable, and when someone asks why the system acted, the record answers.

Set the rules of engagement

Every action clears one authenticated door and answers to your policy — who is acting, on which tenant's data, at what sensitivity — refused by default when in doubt. Enforcement lives outside the agent's own reasoning: a model can be wrong, drift, or swallow hostile instructions and still cannot act beyond the envelope you set.

Give agents real capabilities

Generate media, run arbitrary SQL in a bounded sandbox, forecast time series, solve planning problems, profile datasets, transform local frames, route across governed Earth data, render charts, and drive a live viewer — all under one task, artifact, and policy contract. Adding a hosted or remote server is a catalog registration; the platform stays put.

Share results deliberately

Every output is an owned artifact. Grant read, write, or admin to users and groups; mark an artifact releasable and mint a revocable, expiring link. Downloads pass policy and audit before bytes move.

Command from one console

The operations console opens on the running installation: health, tasks, artifacts, agents, recordings, hosted-server topology, policy, and audit evidence. Domain administration screens manage Map sources, acquisitions, releases, and mobility profiles through the same gateway boundary.

Own the whole installation

Built for contested and disconnected operations: each installation is owned and operated by whoever deploys it — your hostname, your identity provider, your object store. k3d runs the Helm chart locally, fielded Kubernetes runs the same chart in a cluster, and a verified offline bundle runs it fully disconnected.

World Model

A world model of your operational reality

Agent context is a world model: everything the operation senses, remembers, and decides, fused into one durable, queryable model of reality. Sensors and simulators stream the physical world into the record; agents keep their beliefs, missions, and constraints in analytical memory; every decision lands in the log. Each episode assembles a fresh slice of it — the world now (a latest-at snapshot), how it got here (trajectories and history), and what the agent believes and intends — and each act the agent takes becomes part of the model the next episode reasons from. The mission itself lives in the model as data — objectives, constraints, beliefs — so intent is something the agent reasons over, and adaptation stays subordinate to the goal. A mission can arrive in almost any format — a scenario file from another autonomy stack, an exported plan, an annotated map — and the agent structures it into the model: objectives and constraints as rows, geometry through the Map and Frames servers into real coordinates, the original preserved as an artifact.

Context is queried, not accumulated. Because the world model is relational and temporal, an episode's tokens carry answers instead of history: the agent SELECTs exactly the rows it needs from its own memory, pulls a latest-at snapshot or a trajectory from the record, and reads its own past decisions as data. A mission's entire history compresses to the slice that matters this episode.

Data Flow

How data moves

The decision cycle, engineered: Boyd's OODA loop, with orientation living durably in the world model — and one stage Boyd never needed, Account. Observe: producers push into the Recording Hub, and agents query the record back out through governed tools. Act: agents and operators call tools through the gateway; long work becomes a durable task whose completion wakes the sleeping agent. Account: every auth decision, tool call, task transition, artifact grant, and agent step lands in the platform store and the decision log.

Observe
  • Producers push Rerun log streams to one ingest point; the hub persists first.
  • Segments become governed catalog records in the platform store.
  • Agents read slices — a world snapshot, a trajectory — through the recording server.
Act
  • Every tool call enters through the gateway under one flat namespace (sumo__run_batch).
  • Quick tools answer inline; long tools return a durable task handle immediately.
  • Task completion pushes a wake; the next episode assembles with the result.
Account
  • Auth, policy, admin, and task transitions write audit evidence to the platform store.
  • Every decision records what the agent saw, chose, and why — teed into the Rerun decision log.
  • Usage records meter provider spend and tool activity per principal.

Technology Stack

The layers underneath

The capabilities above rest on a small set of layers: an agent runtime, one governed door, a hosted capability plane, shared durable planes, and the recording pipeline. Each layer has explicit contracts, fails closed on invalid configuration, and is exercised end to end by the smoke harness.

Agent Kernel

long-lived autonomous agents in bounded episodes — the Pilot is the first

An agent runs forever because durable stores are the memory: every episode's context is assembled fresh from the agent's memory planes, and everything in flight survives process death. The kernel validates every manifest against its schema before execution. An agent type combines a manifest, a preamble, and SQL migrations.

Wake

A batch of wakes, coalesced and deduped; every wake is persisted with its disposition, so the ledger explains why each episode ran.

Assemble

A fresh slice of the world model — current state, beliefs, and history — plus the wake payload, under a token budget. Truncations point at memory_query.

Episode

One bounded LLM run. Quick gateway tools answer inline; long tools dispatch as durable tasks and keep running. Hooks record every step and terminate over-budget runs.

Persist

Unresolved task descriptors land in the platform store; the episode closes with usage counters and a summary; every step is already in the decision log.

Sleep

Idle on the wake bus. A leased watcher per detached task waits on the result; the heartbeat bounds silence.

↩ wakes: task result · resource change · timer · operator message · elicitation answer — and a wake survives process death: descriptors and watcher leases rehydrate from the platform store on any boot, so kill and restart resumes the same tasks.
Memory planes
  • Scheduling truth — the SurrealDB platform store owns episodes, wakes, task watchers, and leases; the outbox and changefeed deliver wakes across processes.
  • Analytical memory — a sandboxed DuckDB per agent: domain tables (the Pilot: targets, missions, waypoints, constraints, beliefs), a kv store, and an episode-log projection for context assembly.
  • Decision log — an append-only Rerun recording: every turn, tool call, task lifecycle, prompt, and summary, time-indexed and watchable live.
  • The agent reads all three itself: memory_query (guarded SQL), memory_write (bounded mutations, mirrored to the decision log), and timeline_query (dataframe reads over the log).
Durability, proven
  • Detach at episode end: a task's descriptor outlives the process; any later boot rehydrates the live handle and keeps waiting.
  • Token rotation is make-before-break; task ids are principal-scoped, so continuity holds across reconnects.
  • The wake is push, arithmetically: with polling pinned at 60s and the heartbeat at 600s, the smoke's agent slept the task's full ~15s and woke within a second of completion — verified live with a real model end to end.
  • Budget hooks book overruns as budget_terminated, a first-class outcome; hourly windows defer non-priority wakes.
Agents are data
  • At boot, schema validation rejects an invalid manifest. The manifest declares the model endpoint, gateway principal (client-credentials + private-key JWT), episode limits, budgets, schedule, context SQL sections, and writable tables.
  • A preamble and domain migrations complete the profile — cross-episode instructions live here or in memory.
  • Operators reach a running agent through durable wakes: inject a prompt, read status in the console, answer parked tool questions (elicitations).
  • The Pilot rides the operator profile: geodesy inline, planning as tasks, waypoints recorded when plans land.

Shared Planes

the durable foundation every server builds on

Platform Store

One SurrealDB 3.2.0 node (RocksDB) is the durable coordination authority for the whole installation.

  • Owns identity, policy, control-plane revisions, tasks, artifacts, recordings, map catalogs and releases, agents, wakes, audit evidence, and usage
  • The transactional outbox and replayable changefeed carry all cross-process work; LIVE queries accelerate delivery
  • Ordered, versioned migrations; separate admin and runtime credentials
  • Every subsystem uses the same canonical tenant and principal identities

Artifact Plane

A single byte-level policy-enforcement point. Every domain server stores and reads artifact bytes through the plane, calling it with the gateway-signed identity it already holds.

  • S3-compatible object store for bytes; the platform store owns grants, labels, and release state
  • Fresh opaque identity per occurrence: artifact://{uuidv7}; hashes provide integrity and tenant-local dedup
  • Grants to users and groups; revocable expiring links for released artifacts
  • Async producers redeem bounded, expiring write capabilities issued while a live identity was present
  • artifact:// references let one server consume another's output — SQL ingesting a media render, optimization snapshotting to a forecastable table

Contract Core

The provider-neutral contract every server shares: Rust newtypes and enums, exported JSON Schemas, and fail-closed startup validation.

  • Strong ID newtypes: profiles, servers, tools, tenants, scopes, JWT ids
  • Gateway control-plane, auth, and policy config with cross-reference validation
  • Schema-validated deployment profiles: object store, state store, telemetry, network boundaries, regulated-data controls
  • Shared task, artifact, usage, and analytics schemas; JSON Schema export for other languages
  • Gateway-to-server Ed25519 identity assertions — servers hold public verification keys only

Recording Hub

the durable, queryable time-and-space record — one bus, many producers

Every sensor, estimator, and agent streams Rerun data into one ingest point, and the hub persists first: the ingest socket and the durable writer are one process, so every message is fsynced to day-partitioned segment files the moment it arrives. Live files stay readable mid-write and are verified before their optimized replacement at freeze. Every recording and segment becomes a governed record in the platform store, and the recording server answers authorized queries over the same bytes. Files are the record, the hub is the bus, the recording server is the reading room.

Producers → bus
  • Many SDKs stream into one ingest socket, sessioned by recording id.
  • Routing: application-id prefix → dataset (world, agents, …); unmatched → quarantine.
  • Internal-network only; the recording server is the authorized surface.
Durability
  • Segment per (dataset, day, recording), fsynced on write.
  • Footer-less live → decodable to the last whole message after a crash.
  • Freeze runs rerun rrd optimize and verifies before replacing.
Reading room
  • Local QueryEngine is the authoritative file reader.
  • Recordings and segments carry canonical tenant identities in the platform store.
  • latest-at = a world snapshot · range = a trajectory · seal = an artifact.

The datasheet service is the canonical working template: full MCP discovery, direct dataset reads, and resumable profiling on the shared platform store.

The platform's memory. One SurrealDB 3.2.0 node (RocksDB) holding identity, policy, tasks, artifacts, recordings, agents, wakes, audit, and the transactional outbox.

The outbox and changefeed are how work crosses processes durably; every subsystem shares the same canonical tenant and principal identities.

Task Runtimeours

The ledger for long work. UUIDv7 task ids with atomic creation, leases, transitions, cancellation, results, and retention pins in the platform store.

Each operation declares its recovery: resume for deterministic work, webhook_wait for submitted provider jobs, and interrupted_indeterminate for mutating work that fails closed.

Recording Hub — the durable record

Ingest socketOSS

The ingest socket. A gRPC server (re_grpc_server) where every producer pushes data, and which rebroadcasts it live to anyone reading.

It is the single meeting point: producers write to it, viewers read from it. It holds a short in-memory buffer; durability is the writer's job.

Persist-first writerours

Catch-and-persist. Our service that embeds the ingest socket and fsyncs every message it receives to disk before anything else.

Because the socket and the writer are one process, the write is the first thing that happens to arriving data.

Segment file .rrdformat

The record on disk. Rerun's recording format — one growing file per dataset / day / recording.

It stays readable even mid-write, so a crash leaves a decodable file up to the last whole message. This is the durable truth; everything else is a view onto it.

Catalog projectionours

The card catalog. Every recording and segment gets a stable installation identity in the platform store — tenant, owner, classification, labels, state.

Startup reconciles footer-less crash leftovers into explicit records, so the catalog always reflects what is actually on disk.

recording serverours

The reading room. The MCP server that answers authorized questions about the record: discovery, latest-at / range / dataframe queries, subscriptions, and sealing a recording into an artifact.

Access rides the same gateway policy as every other tool, so the record is governed data like everything else.

dataset · recordingconcept

How the record is organised. A recording is one stream (one SUMO run, one agent's decision log); a dataset groups them (world, agents).

A segment's catalog identity is its recording id, so the file on disk and the thing you query are the same identity.

Rerun viewerclient

The window that draws it. A GUI app that renders the streams — vehicles on a map, speeds as graphs, a scrubbable timeline.

It attaches like any other reader: to the ingest socket for a live tail, or to a .rrd file or sealed artifact to replay a whole run. The hosted headless instance is the rerun server above.

GeoPoints · Scalarsformat

Rerun data shapes. Its archetypes carry known semantics — GeoPoints are points with real lat/lon that land on a map; Scalars are named time-series numbers.

Logging data as a known shape is what lets the viewer put vehicles on a map and plot speeds without any per-project wiring.

SUMO Showcase — the live proof

SUMOOSS

The traffic simulator. An open-source microsimulator (Eclipse SUMO 1.27.1) stepping the pinned LuST Luxembourg road network full of individual vehicles through time.

It stands in for a real fleet of sensors: a live world that produces a stream of positions and states to record and reason about.

TraCIprotocol

The socket into the sim. SUMO's Traffic Control Interface — a connection to advance the simulation one step and read or command it.

Through it you get every vehicle's position and speed, and you can change a traffic light or reroute a car mid-run. It stays on the internal network.

sumo-mcpours

The bridge. The one owner of the simulator's control socket: it holds the single serialized TraCI connection, pushes the world into the hub, and exposes SUMO control as governed sumo__* tools on the shared task runtime.

One process is the sole owner of the sim, so reads, pushes, and commands stay ordered.

MCPprotocol

The shared protocol surface. Model Context Protocol carries tools, resources and templates, prompts, completions, tasks, subscriptions, notifications, and structured content with declared schemas between clients, the gateway, and hosted servers.

The gateway governs the complete surface through one profile without replacing URI-based resources or task state with a second tool-only implementation.

Durable taskconcept

A tool call you walk away from. A long operation returns a task handle immediately; the agent sleeps and is woken when the result is ready. The task row, its lease, and its result live in the platform store.

This is how an agent runs a 50-step batch or a signal optimisation while holding zero connections and burning zero episodes waiting.

Resource · subscribeconcept

A condition that wakes you. An MCP resource is a readable endpoint; subscribing to one means the agent is notified the moment it changes.

The showcase exposes sumo://congestion, so a jam forming is a wake the moment it happens.
