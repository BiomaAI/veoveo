Abstract

Persistent autonomy in contested and disconnected operations imposes three requirements that current agent architectures rarely meet together: cognition that outlives any context window, governance that holds even when the model is wrong, and evidence that an independent reviewer can replay. This paper presents the Autonomy Harness, a self-hosted cognition stack that meets all three on infrastructure the operator owns. Agent context is a world model — everything the operation senses, remembers, and decides, fused into one durable, queryable model of reality — from which every reasoning episode assembles a fresh slice. Enforcement lives outside the agent's reasoning: every action clears one authenticated gateway and answers to installation policy, refused by default and audited always. Long-running work executes as durable tasks with declared recovery semantics that survive process death, and every signal the operation produces is persisted before anything else touches it — losslessly, at approximately 225,000 messages per second. A live traffic-world case study closes the perceive–decide–act loop end to end, and the same platform deploys unchanged from a single edge host to an air-gapped cluster.

1. Introduction

Autonomous systems are leaving the laboratory and entering operations: traffic networks, sensor deployments, logistics, field robotics. The intelligence available to run them — large language models and other general reasoners — is powerful, but three properties of real operations expose the gap between a capable model and a deployable autonomous system.

Operations outlast context windows. A mission runs for days or weeks; a model's context holds one conversation. An agent whose knowledge lives in its context forgets its mission at every restart, every upgrade, and every machine failure. Persistent autonomy requires memory that is architectural, not conversational.

Models cannot be their own governors. A reasoner can hallucinate, drift over a long mission, or absorb hostile instructions from the environment it observes. Any property that must hold — who may act, on which data, within which bounds — has to be enforced outside the model's own reasoning, and every enforcement decision has to leave evidence.

Operations cannot depend on someone else's cloud. The environments that most need autonomy — contested, disconnected, regulated — are precisely the ones where a vendor control plane, a required hosted service, or a mandatory network link is a liability. The platform must deploy where the operation is, including behind an air gap, and be owned entirely by its operator.

The Autonomy Harness is a cognition stack built for these constraints. Its thesis: give agents a world model instead of a chat history, govern them from outside their own reasoning, make every piece of long-running work durable with declared recovery, persist every signal before anything else touches it, and package the whole stack so that one operator can run it anywhere. The remainder of this paper develops each of these commitments and validates them on a live traffic world.

2. Contributions

The Autonomy Harness makes six architectural contributions. Each is developed in the section noted and exercised end to end by the validation in Section 10.

  1. The world model as agent context (§4). Everything the operation senses, remembers, and decides fuses into one durable, queryable model of reality. Every reasoning episode assembles a fresh slice of it, so an agent's knowledge survives any restart and outlives any context window.
  2. Cognition governed from outside the model (§6). Every action clears one authenticated gateway and answers to installation policy — refused by default when in doubt. A model can be wrong, drift, or swallow hostile instructions and still cannot act beyond the envelope; even refusals leave evidence.
  3. Persist-first recording (§9). The ingest socket and the durable writer are one process: every message is written before anything else happens to it. Live files stay readable mid-write, segments are verified before replacement, and capture is lossless at approximately 225,000 messages per second.
  4. Durable tasks with declared recovery semantics (§8). Long work detaches from the episode that started it and survives process death. Every operation declares its recovery class — resumable, webhook-bound, or fail-closed — and completion wakes the sleeping agent within a second.
  5. A modular capability plane under one contract (§7). Nine capability modules run at the edge today; any MCP server — local, bridged, or remote — registers in a validated catalog, applies live, and immediately answers to the same task, artifact, and policy contract.
  6. Deployment-invariant cognition (§11). The same platform, the same agents, and the same operational intelligence run on one edge host, across a cluster, sealed inside an air gap, or hybrid — with provenance included and connectivity on the operator's terms.

3. System Model

The system closes one loop. The world — sensors, simulators, and the agents themselves — streams into the record; agents perceive from it, decide, and act through governed tools; their effects land back in the world; and every step deposits evidence. Three flows share this spine. Observe: producers push into the recording hub, and agents query the record back out through governed tools. Act: agents and operators call tools through the gateway; long work becomes a durable task whose completion wakes the sleeping agent. Account: every authentication decision, tool call, task transition, artifact grant, and agent step lands in the platform store and the decision log.

5. The Agent Kernel

An agent runs indefinitely because durable stores are the memory: every episode's context is assembled fresh from the world model, and everything in flight survives process death. The kernel executes typed manifests — an agent type is a manifest, a preamble, and SQL migrations — so fielding another agent is an act of configuration, and one installation runs one agent or a team of them.

StageWhat happens
WakeA batch of wakes, coalesced and deduped; every wake is persisted with its disposition, so the ledger explains why each episode ran.
AssembleA fresh slice of the world model — current state, beliefs, and history — plus the wake payload, under a token budget.
EpisodeOne bounded reasoning run. Quick gateway tools answer inline; long tools dispatch as durable tasks. Hooks record every step and terminate over-budget runs.
PersistUnresolved task descriptors land in the platform store; the episode closes with usage counters and a summary.
SleepIdle on the wake bus. A leased watcher per detached task waits on the result; the heartbeat bounds silence.

Wakes: task result · resource change · timer · operator message · elicitation answer. A wake survives process death — descriptors and watcher leases rehydrate from the platform store on any boot, so kill and restart resumes the same tasks.

5.1 Memory Planes and Durability

Memory planes

  • Scheduling truth. The platform store owns episodes, wakes, task watchers, and leases; the transactional outbox and changefeed deliver wakes across processes.
  • Analytical memory. A sandboxed analytical database per agent: domain tables (the Pilot: targets, missions, waypoints, constraints, beliefs), a kv store, and an episode-log projection for context assembly.
  • Decision log. An append-only recording: every turn, tool call, task lifecycle, prompt, and summary — time-indexed and watchable live.
  • The agent reads all three itself: memory_query (guarded SQL), memory_write (typed mutations), and timeline_query (dataframe reads over the log).

Durability, measured

  • Detach at episode end: a task's descriptor outlives the process; any later boot rehydrates the live handle and keeps waiting.
  • Token rotation is make-before-break; task ids are principal-scoped, so continuity holds across reconnects.
  • The wake is push, arithmetically: with polling pinned at 60 s and the heartbeat at 600 s, a validation agent slept a task's full ~15 s and woke within one second of completion — verified live with a real model end to end.
  • Budget hooks book overruns as budget_terminated, a first-class outcome; hourly windows defer non-priority wakes.
[PRINCIPLE]

Agents are data. A typed manifest — model endpoint, gateway principal, episode limits, budgets, schedule, context sections, writable tables — is validated fail-closed at boot. A preamble and domain migrations complete the profile; cross-episode instructions live there or in memory.

8. Durable Work and Shared Planes

Durable tasks

Every long-running tool runs on the shared task runtime. Task identities are time-ordered; creation, leases, transitions, results, and outbox events are atomic in the platform store. Recovery is declared per operation, so an interruption always has defined, recorded semantics:

ClassMeaning
resumeDeterministic, side-effect-safe work may be reclaimed after its lease expires.
webhook_waitA durably submitted provider job waits for its signed webhook. Completion is webhook-only.
interrupted_indeterminateInterrupted mutating work fails closed and is never replayed automatically.

Shared planes

  • Platform store. One durable coordination authority for identity, policy, control-plane revisions, tasks, artifacts, recordings, agents, wakes, audit evidence, and usage. The transactional outbox and replayable changefeed carry all cross-process work.
  • Artifact plane. A single byte-level policy-enforcement point: object storage for bytes, the platform store for grants, labels, and release state. Fresh opaque identity per occurrence; hashes provide integrity and tenant-local dedup; artifact references let one module consume another's output.
  • Contract core. The typed, provider-neutral contract every module shares, validated fail-closed at startup: strong identities, deployment profiles, shared task and artifact schemas, and gateway-signed identity assertions — modules hold public verification keys only.
[PRINCIPLE]

Two sharing modes, both explicit: authorized grants to users or groups, still constrained by tenant and label policy; and read-only anyone-with-link bearers for artifacts explicitly marked releasable — expiring, download-limited, and revocable.

9. The Record

Every sensor, estimator, and agent streams into one ingest point, and the hub persists first: each message is written — fsynced — to day-partitioned segment files the moment it arrives. Live files stay readable mid-write, so a crash leaves a decodable record, and every segment is verified before its optimized replacement when it freezes. Every recording and segment becomes a governed record in the platform store, and the recordings module answers authorized queries over the same bytes. The record is simultaneously the agents' ground truth and the operation's evidence chain: any run replays, any moment is queryable, and when someone asks why the system acted, the record answers.

10.1 The Loop, Measured

The exact detach → sleep → wake path: the agent calls a task-augmented tool, receives a durable handle immediately, persists the descriptor, ends the episode, and sleeps. The module runs on; completion pushes a wake, and a fresh episode assembles with the result.

PropertyMeasured behavior
Capture throughput~225,000 messages per second, lossless, on the persist-first path.
Crash recoverykill -9 mid-write leaves a decodable segment; restart writes a sibling and never overwrites; startup reconciles leftovers into catalog records.
Wake latencyWith polling pinned at 60 s and the heartbeat at 600 s, the agent slept a ~15 s task in full and woke within one second of completion — push, arithmetically.
Process deathTask descriptors and watcher leases rehydrate on any boot; kill and restart resumes the same tasks with memory intact.

Methodology

  • Smoke harness. End-to-end scenarios with real process orchestration: fake identity provider, provider, and upstream services; a real enterprise IdP realm; live agent sleep/wake; and the full traffic world in a disposable project.
  • Conformance. A generic MCP client exercises the full protocol surface, plus auth-metadata discovery and token tooling; continuous integration runs the full suite, the console build, the IdP integration path, and the deployment contract checks on every change.

11. Deployment Considerations

Deployment is a spectrum, and cognition is constant across it. The same platform runs at the operation's edge on one host, across a cluster, sealed inside an air gap, or hybrid — an edge installation that reaches remote capability when links allow. Wherever it lands, it carries the full stack: one agent or a team of them, every capability module, and the operational intelligence to run the mission — the console, the record, and the audit trail.

FormShape
EdgeOne host at the operation: a single command brings up the full stack behind a loopback edge. Public exposure is the operator's ingress decision.
ClusterThe Helm form: one platform-store StatefulSet, default-deny network policy, optional service-mesh mTLS, separate migration and runtime credentials, operator-supplied telemetry and SIEM wiring.
Air-gapThe verified offline bundle: pinned images, installation material, configuration schemas, checksums, and SBOMs. Installation and verification complete entirely inside the boundary.
HybridAn edge installation that registers remote MCP endpoints and provider-backed modules. The reach is governed like everything else — same policy, same audit, same task contract.

12. Conclusion

The Autonomy Harness closes the loop that persistent autonomy requires. Agents reason from a world model that outlives every context window; they act through a gateway that enforces the operator's policy outside the model's own reasoning; their long work is durable, with recovery semantics declared per operation; and everything the operation senses and decides is persisted first and preserved as evidence. A live traffic world exercises the whole loop — perceive, decide, act, wake — and the same platform deploys unchanged from a single edge host to an air-gapped cluster.

The contracts are the foundation for what comes next: teams of agents coordinating over the same world model, richer capability modules registered from the ecosystem, and fleet-level guarantees expressed over mission outcomes. Each grows on the same spine this paper describes — the world model, the envelope, the durable task, and the record.

[THESIS, RESTATED]

Give agents a world model instead of a chat history. Govern them from outside their own reasoning. Make every piece of long work durable. Persist every signal first. And let one operator own the whole thing, anywhere the mission runs.
