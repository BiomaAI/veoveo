Agent Kernel

long-lived autonomous agents in bounded episodes — the Pilot is the first

An agent runs forever because chat history is never the memory: every episode's context is assembled fresh from the agent's two memory planes, and everything in flight survives process death. The kernel executes typed manifests — an agent type is a manifest, a preamble, and SQL migrations, never kernel code.

Wake

A batch of wakes, coalesced and deduped; every wake is persisted with its disposition, so the ledger explains why each episode ran.

Assemble

State header + the wake payload + manifest-defined SQL sections over the memory database, under a token budget. Truncations point at memory_query.

Episode

One bounded LLM run. Quick gateway tools answer inline; long tools dispatch as MCP tasks and keep running. Hooks record every step and terminate over-budget runs.

Persist

Unresolved task descriptors (serializable) land in the ledger; the episode closes with usage counters and a summary; every step is already in the flight log.

Sleep

Idle on the wake bus. A watcher per detached task holds the blocking tasks/result long-poll; the heartbeat bounds silence.

↩ wakes: task completion push · operator message · heartbeat · resource update · elicitation answer — and a wake survives process death: descriptors rehydrate from the ledger on any boot, so kill and restart resumes the same tasks.
Two memory planes
  • Current truth — an embedded relational database: domain tables (the Pilot: targets, missions, waypoints, constraints, beliefs) plus the kernel ledger of episodes, tasks, and wakes.
  • Flight log — an append-only Rerun recording: every turn, tool call, task lifecycle, prompt, and summary, time-indexed and watchable live in the Recording Viewer.
  • The agent reads both itself: memory_query (guarded SQL), memory_write (typed mutations, mirrored to the flight log), and timeline_query (dataframe reads over the log).
  • The flight log is the log of record: replay rebuilds the domain tables from it alone, so plane drift is recoverable.
Durability, proven
  • Detach at episode end: a task's descriptor outlives the process; any later boot rehydrates the live handle and keeps waiting.
  • Token rotation is make-before-break; task ids are principal-scoped, so continuity holds across reconnects.
  • The wake is push, arithmetically: with polling pinned at 60s and the heartbeat at 600s, the smoke's agent slept the task's full ~15s and woke within a second of completion — verified live with a real model end to end.
  • Budget hooks book overruns as budget_terminated, a first-class outcome; hourly windows defer non-priority wakes.
Agents are data
  • A typed manifest, validated fail-closed at boot: model endpoint, gateway principal (client-credentials + private-key JWT), episode limits, budgets, schedule, context SQL sections, writable tables.
  • A preamble and domain migrations complete the profile — cross-episode instructions live here or in memory, never in a prior prompt.
  • Operators reach a running agent over a loopback channel: inject a prompt, read status, answer parked tool questions (elicitations).
  • The Pilot rides the operator profile: geodesy inline, planning as tasks, waypoints recorded when plans land.

Shared Planes

the shared foundation every server builds on

Artifact Plane

A single byte-level policy-enforcement point. Every domain server stores and reads artifact bytes through the plane, calling it with the gateway-signed identity it already holds.

  • Per-tenant encrypted object store + Postgres grant ledger
  • Every read/write decided by the shared access contract: principal, tenant, data label, grant
  • Immutable content addressing: {scheme}://artifact/{sha256}
  • artifact:// references let one server consume another's output — e.g. SQL ingesting a media render, optimization snapshotting to a forecastable table

Contract Core

The typed, provider-neutral contract every server shares, validated fail-closed at startup.

  • Strong ID newtypes: profiles, servers, tools, tenants, scopes, JWT ids
  • Gateway control-plane, auth, and policy config with cross-reference validation
  • Typed deployment profiles: object store, state store, telemetry, network boundaries, regulated-data controls
  • Shared task, artifact, usage, and analytics schemas; JSON Schema export for other languages
  • Gateway-to-server internal signed identity assertions
